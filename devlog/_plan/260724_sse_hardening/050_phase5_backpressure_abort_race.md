# 050 — Phase 5: Backpressure + abort race (classes 9, 12)

One PABCD cycle. Final hardening: bound the bridge's consumption when the
downstream client is slow, and close the headers-to-reader abort window.
Lands LAST: phases 1-3 pinned the terminal behavior this restructure must
preserve.

IN:

- src/chat/outbound.ts (same discipline in the chat converter)
- src/adapters/run-turn-queue.ts (bounded producer/consumer handoff)

## File change map

### 1. src/bridge.ts — backpressure-aware consumption (~162 start,
### ~438 for-await loop)

Current (verified): `async start(controller)` immediately runs the whole
adapter-event loop and `controller.enqueue(...)` unconditionally; a client
that never reads does not stop adapter consumption (queue grows).

Change (design target — audit may refine mechanics, not the contract):
- Contract: adapter event consumption pauses while the stream's internal
  queue exceeds a high-water mark, and resumes on pull; terminal handling
  (phases 1-3) is unchanged.
- Mechanics: drive the event loop from `pull()` instead of doing all work
  in `start()`. Keep a small internal buffer so a single pull can emit the
  frames generated by one adapter event (lifecycle pairs must not split
  across pulls in a way that reorders events — FIFO is sufficient).
- heartbeat/stall timers keep running regardless of pull state: a stalled
  upstream must still time out even if the client is not reading (the
  synthesized incomplete goes into the bounded buffer; if the buffer is
  full AND the client never reads, the stream is cancelled and the
  upstream reader is cancelled too — no unbounded growth).
- High-water mark: constant (e.g. 256 enqueued chunks) documented inline;
  no config surface in this phase.

### 2. src/chat/outbound.ts — same contract (~140)

The converter wraps a ReadableStream around the Responses SSE generator;
apply the same pull-driven discipline (it sits downstream of the bridge,
so with 1 in place its queue is the only remaining unbounded one).

### 3. src/adapters/run-turn-queue.ts + core.ts runTurn wiring (A-gate
### finding, High)

Current (verified): core.ts:1153/1176 starts `void runTurn()` before the
bridge consumes anything; run-turn-queue.ts:57 pushes into an unbounded
array when no consumer reads. Bridge-side backpressure alone cannot fix
this producer.

Change:
- Give the queue a high-water bound: when the internal array exceeds the
  bound AND no consumer is attached/reading, the producer side observes
  backpressure (await consumer drain) or, past a hard cap, the turn's abort
  fires and the queue terminates with an error event — never unbounded
  growth.
- Wire queue termination to the same abort the phase-2 terminal path uses,
  so first-terminal stops the producer here too.
- Exact bound constant documented inline; no config surface.

### 4. src/server/responses/core.ts — abort race (~1300 fetch -> ~1473
### reader)

Current (verified): the generic adapter path attaches no guard between
fetch resolution and reader creation; abort.ts:126 documents the Bun
rejection race and exports cancelBodyOnAbort (:137), used today only by
web-search executors.

Change:
- Route the adapter fetch through cancelBodyOnAbort (or the established
  equivalent in that file) so a client abort landing in the
  headers-to-reader window cancels the body on the awaited path instead of
  producing an unhandledRejection.
- Match the existing web-search usage pattern; no new helper unless the
  shapes differ.

### 5. First-frame latency verification (A-gate)

Pull-driven consumption must not delay the first frame for consumers that
read immediately: verify Claude outbound (claude/outbound.ts:355) and the
WS pump (ws-bridge.ts:179) still observe first bytes without an extra
event-loop turn vs the eager baseline (bridge-live-delivery.test.ts is the
anchor; add a first-frame assertion if missing).

## Accept criteria + activation scenarios

1. Slow-consumer: a bridge stream whose reader is never pulled does not
   consume more than (high-water + in-flight) adapter events within a
   probe window (baseline probe consumed 77 events/100ms). Activation:
   scripted event generator + unread stream; assert consumption count.
2. Resume: after reads begin, buffered frames drain FIFO and the stream
   completes with the same event sequence as an unconstrained run
   (ordering pin against bridge.test.ts expectations).
3. Stall while unread: with a shortened stall budget and a never-pulled
   stream, the synthesized incomplete/cancel path runs and the upstream
   generator is cancelled (no leak).
4. Abort race: abort fired between header resolution and reader attach
   produces no unhandledRejection and cancels the body. Activation:
   fault-injection test mirroring abort.ts:126's documented race.
5. runTurn unread-reader: a Cursor-style queue with no consumer stops
   growing at the bound; abort fires at the hard cap; with a consumer,
   ordering matches the unbounded baseline.
6. Live-delivery: bridge-live-delivery.test.ts stays green (pull-driven
   must not reintroduce event-loop coalescing for an actively-reading
   client); Claude-outbound and WS first-frame checks pass.
7. Full gate: `bun run typecheck` + bridge/chat/passthrough suites green.

## Risks

- This restructures the hottest loop in the proxy; the mitigation is that
  phases 1-4's behavior is pinned by then-existing tests, and
  bridge-live-delivery guards latency regressions.
- pull-driven streams change cancellation timing; cancel-drain behavior is
  pinned by relay-eager/passthrough-abort tests for the native side and
  must be matched here.
